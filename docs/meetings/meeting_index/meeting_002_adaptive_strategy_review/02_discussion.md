# 讨论记录：自适应策略评审会

## 议题：单一算法（方案 A）vs 桶大小自适应多路径

**参照数据（POC v2 实测）：**

| 方案 | 1M uniform | 5M uniform | 10M uniform | 10M skewed |
|------|-----------|-----------|-------------|------------|
| A: AVX2 SIMD二分 | 154.4 | 145.8 | 171.9 | 172.4 |
| B1: high16dir+lo16 | 75.0 | 220.9 | 442.3 | 1267.1 |
| B2: high16dir+AVX2二分 | 122.7 | 210.4 | 268.3 | 216.0 |

---

## 第一轮：各 Agent 立场陈述

### 1. 架构项目经理

**立场：反对自适应多路径，坚持方案 A。**

核心论证：

1. **两轮 POC 的核心教训被忽视了**：三次重大预测错误（"10M 仅 1.03x"、"B 预计 2.9x-3.6x"、"B2 预计 3.4x"）的共同原因是 SIMD 微观性能无法纸面建模。自适应策略是纸面分析逻辑的进一步延伸，在零 POC 验证的情况下做决策是重蹈覆辙。

2. **自适应在大数据量场景实质上就是 B2**：10M 均匀数据桶均 153 元素，绝大多数查询走"大桶 AVX2 二分"路径。但该路径 = dir 定位 + AVX2 二分——正是 B2，实测始终比 A 慢 25%-56%。

3. **倾斜数据灾难仍然存在**：skewed 下 80% 数据挤入同一桶，mid8 二级目录未经 POC 验证，且从 B1 skewed 1267 cy 的表现看，多一层 mid8 无法逆转线性退化。

4. **1M 的 2.1x 是工程经济学上的伪命题**：154 vs 75 = 23ns 绝对差距。用 50% 额外内存、5x 代码量、2 个 magic number、72+ 测试场景去换 23ns，不成立。

5. **替代方案**：如果用 `n <= 2M` 判断做模块级分发（而非嵌入核心循环的三路分支），复杂度在调用层，解耦、可独立测试、可独立删除。

---

### 2. 算法工程师

**立场：坚持方案 A，不引入自适应多路径。**

核心论证：

1. **盈亏平衡点拟合**：从实测数据反向拟合，`T_small ≈ 64`（理论值约 50-55，实测因 cache 效应偏高）。桶内 ≤64 元素时 lo16 扫描更快，>64 时 AVX2 二分更优。

2. **mid8 二级目录分析**：理论上可节省 ~15 cycles，但仅对 800 万元素的极端倾斜桶有效。在 uniform 10M 下，mid8 无触发机会，且 dir 税仍存在。

3. **自适应策略的本质矛盾**：
   - 10M uniform（最常见场景）：桶均 153 > T_small，走"大桶 AVX2 二分"路径 = B2 = 268 cy，比 A 的 172 cy 慢 56%
   - 1M（唯一胜利场景）：桶均 15 < T_small，走 B1 = 75 cy，比 A 快 2.1x
   - **净效果：最常见的生产场景变慢，边缘场景有改善。这是糟糕的 trade-off。**

4. **承认之前错误**：两轮 POC 中我三次预测被实测推翻。教训：简单的直接路径 > 精巧的多层间接路径。这次我不再犯同样的错误。

### 3. 后端工程师

**立场：坚定方案 A（单一算法）。自适应多路径策略在工程上不值得。**

核心推导：

| 场景 | 桶均元素 | 自适应路径 | 实际性能 | vs 方案A |
|------|---------|-----------|---------|---------|
| 1M uniform | ~15 | B1（小桶扫描） | ~75 cy | **自适应胜** |
| 5M uniform | ~76 | B2（AVX2二分） | ~210 cy | **慢 44%** |
| 10M uniform | ~153 | B2（AVX2二分） | ~268 cy | **慢 56%** |
| 10M skewed | ~152/800万 | B2（巨桶二分） | ~216 cy | **慢 26%** |

关键洞察：**在 5M+ 的三个场景中，自适应选中的"最优路径"就是 B2，而 B2 始终比 A 慢。** 自适应策略的所有大桶场景都走 B2 路径，所以自适应在 5M+ 时必定慢于 A。

工程复杂度税：

| 维度 | 方案 A | 自适应策略 | 差距 |
|------|--------|-----------|------|
| 代码行数 | ~120 | ~270 | 2.3x |
| 搜索代码路径 | 1 | 3 | 3x 测试面 |
| 数据结构 | 1 数组 | 3 数组 | 3x 内存管理面 |
| 需要调优的参数 | 0 | T_small + T_huge | 微架构耦合 |

**反建议**：如果非要做小数据集优化，用标量线性扫描 + dir（不要 lo16 数组），代码 10 行，内存仅增 262KB。但即使如此，也不值得——因为 1M 场景在目标应用中概率太低。

### 4. 代码安全专家

**立场：方案 A 是更安全、更正确的选择。** 发现一个潜伏的 high16 dir 构建 bug。

安全/正确性对比：

| 维度 | 方案 A | 自适应策略 | 结论 |
|------|--------|-----------|------|
| 数据结构数量 | 1 | 3 | A 更少攻击面 |
| 初始化哨兵陷阱 | 无 | **存在** (`prev_h=0xFFFF` bug) | A 优 |
| 中间索引依赖链 | lo/hi 直接边界 | target→dir→bucket→search（3 层） | A 优 |
| COW 一致性 | 单指针天然安全 | 三指针需 struct 级原子保证 | A 优 |
| 已验证查询量 | ~16M 次零错误 | 仅 benchmark 验证 | A 优 |
| 搜索核心代码行数 | ~35 | ~120+（含构建） | A 优 |

**发现的 high16 dir 构建 bug**（[poc_benchmark_v2.c:L80](file:///c:/Users/Administrator/Documents/trae_projects/Int32_search_algorithm/src/poc_benchmark_v2.c#L80)）：
```
prev_h = 0xFFFF 初始化：
当全部元素落在 [-65536, +∞) 时，首个元素 high16 恰好 = 0xFFFF，
h == prev_h，dir[0xFFFF] 永远不会被赋值为 0，
backfill 后对该桶的查询全部返回"不存在"（静默假阴性）。
```

**三数组 COW 风险**：若 COW 实现以三个独立指针分别交换（而非单 struct 原子交换），Reader 会读到跨代混合数据，导致隐蔽的一致性问题。

**结论**：为 1M 场景的 22ns 优势接受这些正确性风险，不符合安全工程原则。

### 5. 全栈开发工程师

**立场：坚定方案 A。**

核心论证：

1. **API 设计**：方案 A 三函数，零决策暴露。自适应方案的 `create` 阶段构建开销（dir + lo16，O(n) 扫描 + O(n) 拷贝）对调用方不可预测。如果暴露 flags 控制，接口变复杂。

2. **ROI 分析**：为 22ns 付出 50% 额外内存、3x 代码量、倾斜下 7.4x 退化，不划算。

3. **长期维护的"组合爆炸"**：
   - 方案 A 加 AVX-512：替换 3-4 行 intrinsic
   - 自适应加 AVX-512：每条路径 × 每种桶大小 = N×M 测试矩阵
   - 再加布隆过滤器：决策树翻倍
   - **单一方案 O(n)，自适应方案 O(2^n)**

4. **D-003 精神冲突**：D-003 全票决议 "不再使用低 N 位作为 key"。自适应策略的 high16 dir + lo16 scan 本质上仍是分位策略，是用"high16 作为桶号 + lo16 作为桶内 key"换了个壳。这是倒退到已被 POC 证伪的方案。

5. **建议**：如果未来有强烈的小数据量优化需求，做一个独立的 `is32_search_small` 变体（完全独立的代码路径），不污染主路径。

## 交叉讨论

由于首轮 5/5 Agent 一致反对自适应多路径，所有核心论点高度一致，无需进入正式交叉讨论轮次。以下是各 Agent 观点的交叉印证摘要：

| 共识点 | 支持 Agent | 核心理由 |
|--------|-----------|---------|
| 自适应退化到 B2，最常见场景更慢 | 架构师、算法、后端 | 10M 均匀数据走大桶路径 = B2，慢 56% |
| 1M 的 22ns 绝对差距不值 | 架构师、后端、全栈 | 23ns < 一次 DRAM 访问，+50% 内存不划算 |
| 组合爆炸阻断未来扩展 | 架构师、全栈 | AVX-512/布隆 线性叠加 vs 指数爆炸 |
| 安全/正确性风险 | 安全专家 | prev_h bug + 三数组 COW + 更少 POC 验证 |
| D-003/D-009 精神一致性 | 全栈、架构师 | 自适应 = 回到已舍弃的分位策略 |

一致结论：方案 A（AVX2 SIMD 二分）为唯一查询路径。自适应多路径不予引入。

## 唯一未决问题：B1 的 1M 胜利如何处理

**讨论共识**：记录为已知 trade-off。方案 A 在 <2M 数据规模时离理论最优有约 2x 差距（绝对值 22ns），考虑代码简洁性和数据分布鲁棒性，不引入特殊处理。

**未来扩展建议**：如果用户反馈要求小数据量极致性能，采用模块级分发（`n <= 2M` 独立代码路径），而非嵌入核心循环的自适应逻辑。

## 第三轮（人工修正方案）：构建时一次性选路

### 方案定义

人工提出修正："自适应是邪路。分完桶看 max/min bucket size，数据均匀就全走 B1，否则全走 A。建立两条完整路径，构建时选一条。"

**核心区别 vs 已否决的"每查询自适应"**：

| 维度 | 每查询自适应（已否决） | 构建时一次性选路（新） |
|------|----------------------|---------------------|
| 决策时机 | 每次查询判断 bucket_size | `create()` 时一次性判断 |
| dir 定位开销 | 每次查询支付 ~16-20 cy | 选 A 则 dir 已释放（0 cy）；选 B1 则本身包含 |
| 热路径分支 | `if/else` 在搜索循环内 | 函数指针 dispatch（单次） |
| 最差退化 | B2 慢 56%（268 cy） | A 路径（172 cy），不会更差 |
| 代码路径 | 3 条交织（小/大/巨桶） | 2 条独立正交路径 |
| 扩展复杂度 | O(2^n) | O(n) |

### 各 Agent 评估

#### 架构项目经理：有条件批准

**核心判断**：新方案在架构层面避免了旧方案的全部三项根本缺陷（dir 税、热路径分支、组合爆炸）。此为 Factory Pattern + Strategy Pattern 在构造时绑定，非运行时多态。

**六项准入条件**：

| 编号 | 条件 |
|------|------|
| C1 | D-003 条件豁免正式记录（B1 仅在分布验证后启用） |
| C2 | 阈值硬编码 + 调试日志记录 P90 桶大小 |
| C3 | B1 路径 COW 需 struct 级原子交换 |
| C4 | prev_h=0xFFFF bug 必须修复 |
| C5 | dir 分配失败时静默回退到 A 路径 |
| C6 | handle 中标记策略（STRATEGY_A / STRATEGY_B1），零额外分支 |

---

#### 算法工程师：参数需修正

**盈亏平衡点精确拟合**：

| 数据量 | 桶均 | 加权平均 | B1 预估 | A 预估 | 差值 | 推荐 |
|--------|------|---------|---------|--------|------|------|
| 1M | 15 | 16 | 75 cy (实测) | 154 cy (实测) | **-79** | B1 |
| 2M | 30.5 | 31.5 | ~110 cy | ~148 cy | **-38** | B1 |
| 3M | 45.8 | 46.8 | ~150 cy | ~147 cy | **+3** | 平手 |
| 4M | 61.0 | 62.0 | ~187 cy | ~146 cy | **+41** | A |

**关键修正**：
- T_small 应设为 **45-50**（非 64）。64 偏乐观，4M 时 max_bucket≈61<64 会误选 B1。
- **单用 max_bucket 不够**。应用 `weighted_avg = Σ sz² / n`（查询视角期望桶大小）。
- 推荐决策规则：`IF max_sz > 0.1×n → PATH_A`（防倾斜）；`ELIF weighted_avg ≤ 45 → PATH_B1`；`ELSE → PATH_A`。
- **必须 POC 验证** 2M/3M/4M crossover 点后敲定阈值。承认纸面模型在 SIMD 微观性能领域的历史准确率不足 50%。

---

#### 全栈工程师：通过

- API 零影响，`create()` 延迟差异淹没在排序 O(n log n) 中。
- D-003 是**精细化（refinement）而非违背**："禁止无条件分位" → "分布验证后条件性启用"。
- 建议 cutoff 约 2M-2.5M，需 POC 校准。
- 两条路径独立演进：A 加 AVX-512（3 行），B1 加 AVX-512 lo16 宽化（2 行），零耦合。

---

#### 代码安全专家：仍劣于纯 A，但远优于旧方案

**风险矩阵变化**：

| 风险 | 旧方案 | 新方案 | 变化 |
|------|--------|--------|------|
| prev_h bug | HIGH | HIGH | 未变（需修复） |
| COW 三数组 | HIGH | HIGH→MEDIUM | 仅 B1 路径且需 struct 级交换 |
| 三路分支测试爆炸 | HIGH | **消除** | 查询时不分支 |
| B2 退化 | MEDIUM | **消除** | 不选 B2 |
| mid8 复杂度 | MEDIUM | **消除** | 不引入 |
| 分布分析误判 | — | MEDIUM（新引入） | 可低成本缓解 |

**底线要求**：
- prev_h bug 修复为**前置条件**（无此修复，所有 dir 方案均不安全）
- 一致性校验内置：`dir[0]==0, dir[65536]==n, dir` 单调不递减。失败回退 A。

---

#### 后端工程师：推荐（带约束）

- 新增代码 ~102 行（vs 纯 A ~120 行，总计 ~222 行，约 1.85x）。
- 构建代价（~0.3ms @ 10M）完全可忽略，淹没在排序 O(n log n) 中。
- B1 内存增量自限在 ≤ 8.4MB（@4M 触发上限）。
- **但如果目标场景严格 >5M**，方案退化为纯 A + 浪费 0.3ms dir 构建，不值得。鉴于目标"百万到千万"，推荐实施。
