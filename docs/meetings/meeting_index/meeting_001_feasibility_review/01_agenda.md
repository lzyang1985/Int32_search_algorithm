# 会议议程：Int32查找算法方案可行性评审会

## 1. 项目背景

构建一个高性能 Int32 精确查找算法库，目标数据规模 **100万~1000万行**，支持 Windows 和 Linux 平台。

## 2. 议题：技术方案可行性评审

讨论以下架构方案的可行性、潜在风险及优化建议：

### 2.1 底层数据结构：有序链表

- 链表便于插入/删除操作
- 查找前需保证链表有序
- **讨论点**：百万级数据下链表的排序开销、内存碎片、缓存不友好问题

### 2.2 一级索引：平行数组 + AVX 加速

- 一侧存放数据的低8位或低16位（作为 SIMD key）
- 另一侧存放对应数据的位置指针
- 可塞入 AVX 寄存器进行向量化比较
- **讨论点**：低8位 vs 低16位的选择策略；AVX2 vs AVX-512 的兼容性；非对齐访问处理

### 2.3 二级索引：间隔抽样

- 从一级索引中以固定间隔抽取，形成二级索引
- **讨论点**：抽样间隔的选择；二级索引的存储结构

### 2.4 稀疏目录（Sparse Directory）

- 覆盖整个 Int32 范围（约 42 亿），每隔约 1000 位抽取
- 直接指向二级索引区域
- 定位查找范围的起始和结束位置
- **讨论点**：稀疏目录的大小（约 400 万项）；内存占用权衡

### 2.5 桶排序 + 布隆过滤器（外层）

- 桶排序降低单序列大小
- 每桶可选布隆过滤器，快速排除不存在的 key
- **讨论点**：桶数量选择；布隆过滤器的假阳性率和参数配置

### 2.6 多线程支持

- **讨论点**：读多写少场景下的并发策略；锁粒度设计（per-bucket锁？RCU？）

### 2.7 平台兼容性

- Windows (MSVC/MinGW) + Linux (GCC)
- AVX intrinsic 的跨平台封装

## 3. 关键决策点

1. 链表 vs 其他数据结构（如有序数组 + gap buffer）的取舍
2. 一级索引 key 用低8位还是低16位？
3. 稀疏目录的实际内存开销是否可接受？
4. 多线程并发模型如何选择？
5. 布隆过滤器是否作为可选特性？

## 4. 预期产出

- 方案可行性结论（通过/有条件通过/需重新设计）
- 模块划分与子项目拆分方案
- 总需求文档草稿
- 技术路线文档草稿
