# 讨论记录：实施方案讨论会

## 与会 Agent 意见汇总

### 议题 1：模块划分

| Agent | 核心观点 | 模块数量 |
|-------|----------|----------|
| **架构师** | 4 层结构：平台抽象层 (PAL) → 构建与选路层 → 查询引擎层 → 公开 API 层 + 可选扩展层，每层有明确接口契约 | 15+ 文件，按层分目录 |
| **算法工程师** | 6 个独立可测试模块，按"构建时一次 vs 查询时百万次"分割。模块 D (search_avx2) 零依赖最先实施 | 6 模块，按生命周期分 |
| **后端工程师** | `src/core/` + `src/build/` + `src/platform/` + `src/util/` + `api.c`，单一公开头文件 `include/int32_search.h` | ~12 文件，按职责分目录 |
| **全栈工程师** | 4 层：核心数据结构 → 查询引擎 → 构建选路 → 平台适配，单仓库结构，预估 ~800 行 + ~200 行头文件 | 14 个 .c 文件 |
| **安全专家** | 关注安全边界：外部输入、COW 并发、SIMD 内存访问。建议 dir_validate 作为强制门控而非调试辅助 | 不涉及文件组织 |

**共识**：
- 所有 Agent 同意按职责分层，搜索核心与构建逻辑分离
- 公开头文件单一入口（不透明句柄）
- 内部头文件不安装

**差异点**：
- 后端工程师建议细粒度目录（`src/core/`、`src/build/`、`src/platform/`），全栈工程师倾向更扁平的结构
- 安全专家独立提出 `block = hi - 8` 下溢风险和 COW 内存序要求

### 议题 2：子项目拆分

| Agent | 立场 | 方案 |
|-------|------|------|
| **架构师** | 拆分为 3+1 | libint32search + benchmark + test_suite + poc_archive |
| **算法工程师** | 必须拆分 | 核心库 + benchmark 独立编译，benchmark 链接核心库验证 API 契约 |
| **后端工程师** | 三层目标拆分 | libint32search.a + test_runner + bench_runner，CMake 三层 |
| **全栈工程师** | 不拆子项目 | 单仓库多编译目标（lib/test/benchmark/example），一个 Makefile |
| **安全专家** | 不独立安全子项目 | 安全测试贯穿所有阶段，各阶段有特定安全关注点 |

**分析**：后端的"三层目标"与全栈的"多编译目标"本质相同——都是单一代码库产出多个构建产物。架构师的 3+1 和算法工程师的 split 同样是同一思路的不同表述。**实无根本冲突。**

**共识**：
- 单仓库，产出一个主库（libint32search）+ 测试可执行文件 + benchmark 可执行文件
- 不拆成多个需要独立管理版本号的子仓库

### 议题 3：实施顺序

| Agent | 阶段划分 | MVP 范围 |
|-------|----------|----------|
| **架构师** | P0(MVP A路径) → P1(B1+COW) → P2(扩展+跨平台) | 仅 Path A |
| **算法工程师** | MVP=模块 A+D → B1 链路增强 → AVX-512/Bloom 独立 patch | 模块 A+D (build_sorted + search_avx2) |
| **后端工程师** | M0→M1/M2/M3并行→M4(AVX2)→M5(API)→M6-M8(扩展) | M0+M1+M2+M3+M4+M5 |
| **全栈工程师** | 阶段1 MVP Week1 → 阶段2 v1.0 Week2 → 阶段3 v1.1 Week3 | 仅 Path A + 标量回退 |
| **安全专家** | 安全左移：标量引用先行 → ASan/UBSan 同步交付 → dir_validate 强制门控 | 建议先实现标量引用实现 |

**强烈共识**：MVP = Path A only（AVX2 SIMD 二分）。B1 不在 MVP 范围。

**安全专家的关键建议**（被所有 Agent 认可）：
1. 先实现标量二分作为黄金基准
2. ASan/UBSan 与核心代码同步交付
3. dir_validate 不仅调试使用，生产构建也保留
4. COW 需要明确内存序要求

## 交叉讨论：三项需要确认的分歧

### 分歧 1：文件目录组织粒度

- 后端工程师：`src/core/`、`src/build/`、`src/platform/`、`src/util/` 四个子目录
- 全栈工程师：单层 `src/` 目录下直接放文件

**决议方向**：考虑到项目总代码量 ~1000-1500 行，细粒度目录过度工程化。建议折中：`src/` 下不分子目录，通过在文件名前缀区分层级（如 `search_avx2.c`、`build_dir.c`、`platform_cpu.c`）。头文件 `include/` 下仅一个公开头文件。

### 分歧 2：SIMD 多版本实现策略

- 后端工程师：同一 search_a.c 编译 4 次传不同宏（-DINT32_SEARCH_AVX2 等）
- 架构师/算法工程师：AVX2 主力先行，SSE2/AVX-512 正交补充

**无实质冲突**——两者都同意 AVX2 先行。编译策略的技术细节可在 Design 阶段细化。

### 分歧 3：是否立即生成总需求文档和技术路线文档

- 规范要求：会议达成一致后由文档管理 Agent 生成这两份文档
- 实际情况：本次会议为实施方案讨论，更接近 Architect 阶段的输入

**决议方向**：本次会议后应生成这两份"宪法级"文档。总需求文档基于会议讨论结果提炼，技术路线文档直接采用会议决议中的方案全貌。
