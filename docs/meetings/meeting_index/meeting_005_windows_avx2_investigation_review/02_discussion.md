# 讨论记录 — Windows AVX2 异常调查结果审查会

## 议题 A-01：调查方法论审查

### 架构项目经理

调查方法论**基本充分**，INVEST-01~08 构成系统性的假说排除链路，四个假说（popcount 软件模拟、CPU 检测假阴性、预热不足、计时精度）均被干净排除。但根因分析存在**逻辑不一致**：报告声称 ~12 cy/iter 是 AVX2 减速的根本原因，然而 Linux 上同一算法的 AVX2 达到 168 cy/q (5.26x)，反算每迭代约 10 cy，与 Windows 的 ~12 cy 理论值接近。如果 CPU 指令延迟是固有瓶颈，Linux 也应该慢。实际 Windows AVX2 约 951 cy/q，反算 ~56 cy/iter（远超 12 cy），指令延迟分析只能解释 ~13% 的差距，**剩余 87% 未解释**。

### 算法工程师

**强烈附议**。POC v1 在**同一台 Windows 机器**上达到 120 cy/q (3.53x)，与当前 951 cy/q 差了 **8 倍**。这直接驳斥了"12 cy/iter 是算法固有瓶颈"的结论。真正的盲区是**编译选项差异**：POC 使用 `-march=native`（启用 Skylake 特定调度 + 循环对齐 + DSB 优化），当前项目只用 `-mavx2`（generic 调度）。推荐立即执行 `-march=native` vs `-mavx2` 对比实验（E1-E4），耗时约 30 分钟即可定位根因。

### 代码安全专家

调查方法论**合规**，但汇编分析范围不足（仅 6 条核心指令，未检查序言/结语中是否有 YMM 寄存器溢出）。建议补充 `objdump -d` 完整反汇编分析。此外，`vals[last_le]` 的边界访问虽经独立推导确认安全，但调查应将其纳入显式检查清单。

### 后端工程师

Benchmark 改造质量**优异**，四组对照的"洋葱剥皮法"设计正确，`__rdtscp()` + `_mm_lfence()` + ~500ms 预热已达到工业级标准。唯一缺憾：未报告标准差/多次运行范围，建议补充 N=5 重复测量。

> **A-01 共识**：调查方法论充分，但根因分析不完整。关键盲区：未对比 `-march=native` vs `-mavx2`，未解释同平台 POC (120 cy/q) vs 当前代码 (951 cy/q) 的 8x 差异。

## 议题 A-02：根因分析确认

### 算法工程师

报告的理论框架（SIMD 宽度 × 迭代延迟 vs 迭代减少数）方向正确，但结论"AVX2 只会在大规模数据（N >> 100M，cache miss 主导时）才有优势"**过于悲观**。实际上 POC 实测数据表明，当编译器优化到位时 AVX2 在**所有规模**上都显著优于标量（1M: 3.53x, 5M: 3.66x, 10M: 5.09x）。微架构因素补充：

1. **指令对齐和 DSB 缓存**：`-march=native` 启用循环 32B 对齐，适配 Skylake DSB（uop cache）；`-mavx2` 可能未对齐导致 DSB miss 回退 MITE（传统解码器），引入前端瓶颈。
2. **端口 0 串行化**：`vpcmpgtd`→`vmovmskps`→`popcntl` 是严格串行依赖链，无并行机会，但这与编译选项无关。
3. **`_mm256_castsi256_ps`**：零延迟寄存器重命名，跨域惩罚仅在 `vmovmskps` 处发生（~1 cy），注释准确无误。

### 架构项目经理

同意算法工程师的修正。真正的根因应重新定性为"编译器代码生成质量差异"，而非"算法固有的性价比边界"。在获得 `-march=native` 对比实验数据之前，报告 §7.1 的结论应标注为"暂定假说"。

> **A-02 共识**：根因分析方向正确但结论过于悲观。实际根因更可能是编译选项不足（`-mavx2` vs `-march=native`），而非算法固有瓶颈。需通过编译选项对比实验最终确认。

## 议题 A-03：短期处理策略

### 架构项目经理

**不同意"默认走标量"**。Linux 上 5.26x 加速比是已验证的真实数据，AVX2 路径不应降级。推荐：标记已知限制 + 保留 AVX2 路径 + Phase 2 构建时 microbenchmark 自检。

### 后端工程师

**强烈同意"构建时决策"方案**。具体实现：
1. 在 `int32_search_impl_t` 中增加 `search_fn` 函数指针字段
2. `create()` 中一次性完成 CPU 检测 + 路径决策
3. `find()` 简化为 `return impl->search_fn(vals, n, key, out_index)`
4. 阈值 `INT32_SEARCH_AVX2_MIN_N = 10,000,000` 定义为编译期常量
此方案将消除每次查询 `platform_cpu_has_avx2()` 的冗余调用（省 ~8-15 cy/q）。

### 代码安全专家

如果实施策略变更，必须注意：
1. `platform_cpu_has_avx2()` 语义不变（保持纯硬件检测），调度逻辑放在 `create()` 中
2. CI 中保留 `FORCE_AVX2` 编译配置，防止 AVX2 代码腐烂
3. API `else` 分支（标量回退）缺乏在 AVX2 机器上的覆盖测试

> **A-03 共识**：AVX2 路径保留，不默认走标量。采用后端工程师提出的"构建时决策"方案（函数指针 + N 阈值）。Linux 上 AVX2 5.26x 性能已验证，不应因单一编译器平台表现而降级。

## 议题 A-04：第四波 Linux 对比

### 算法工程师

Linux 对比有价值但**优先级应降级**。POC 已证明同一 Windows 平台可达 3.53x，跨平台对比无法回答"为什么同一平台差 8x"。真正应优先做的是本平台的 `-march=native` vs `-mavx2` 实验（E1-E4，30 分钟可完成）。Linux 验证在完成上述实验后退化为 P3。

### 后端工程师

Linux vs Windows 的 ABI 差异微小（Red Zone、调用约定等），预计 Linux 上加速比在 0.50x-0.60x（vs Windows 0.45x-0.55x），**不会逆转方向**。建议做 Linux 验证（P2），但不阻塞 Phase 1 收尾。

### 架构项目经理

原 FIXPLAN 第四波优先级 P2 → 建议**因算法工程师的发现而降级为 P3**。但**新增 P0**：本平台 `-march=native` vs `-mavx2` 对比实验。

> **A-04 共识**：Linux 对比降级（P2→P3），新增 P0 任务：本平台编译选项对比实验。原因：POC 已证明同一硬件可达 3.53x，跨平台对比无法定位根因。

## 议题 A-06：Phase 2 自适应策略影响

### 算法工程师

当前自适应策略（D-015，仅基于数据分布）不受本次发现影响。但需增加**Level 1 编译时决策**（`-march=native` 检测）和 **Level 2 运行时决策**（仅当 Level 1 不可用时：N × sizeof > 2 × L3 → AVX2）。

### 架构项目经理

Phase 2 构建时选路框架不受根本影响，但需增加编译器质量维度作为决策输入（构建时 microbenchmark 自检，约 10ms）。

> **A-06 共识**：Phase 2 自适应策略方向不变，但需增加编译时/构建时的编译器质量检测维度。
